# Decisions log

| Date | Decision | Rationale |
|------|----------|-----------|
| — | Split project into two games | Each can go deep on its strength; smaller scope per game; staged release possible |
| — | Multiplayer required for both | Core to the design vision |
| — | Player-built ships in Game B use modular free-placement | Like The Forest / survival games, not voxel |
| 2026-05-27 | Buoyancy: weight vs. score (not distributed physics) | Cheap to network, no exploding physics, legible to players. Standard pirate-sim approach (Sea of Thieves, Subnautica, ARK). |
| 2026-05-27 | Ship pieces are data, not per-frame physics samples | Game B hulls aggregate piece volumes/masses into the buoyancy score; wave sampling stays at tier points. Resolves the voxel-resolution question. |
| 2026-05-27 | Sample-point tier system, rectangular grid, Tier 2 default | Ships are anisotropic + port-starboard symmetric → rectangular maps cleanly. Tier 2 (3×3 = 9) captures pitch + heel. Tier auto-assigned by size/piece count. |
| 2026-05-27 | Performance target: 2016 mid-range PC (i5-6500 / GTX 970-class / 8GB / 1080p) | Reference bar: Sea of Thieves on Xbox One. Forces practical choices (Gerstner over FFT, cube-map over SSR). |
| 2026-05-27 | Wave generation: multi-frequency Gerstner (FFT cut for v1) | Fits 2016 perf target; analytic on CPU for cheap buoyancy sampling. Revisit FFT only if hero shots demand it. |
| 2026-05-27 | Platform: Steam first | Steamworks gives SDR + Lobbies + Cloud + Workshop + Voice. $200 + 30%, no infra costs. |
| 2026-05-27 | Multiplayer: P2P session-based via Steamworks | No persistent shared world. Replicate world feel via Sea of Thieves design tricks (voyage events, leaderboards, crossover encounters). |
| 2026-05-27 | Ship sharing: Steam Workshop, versioned + validated | Game B → Game A pipeline rides Workshop. File format must be Workshop-compatible. |
| 2026-05-27 | **REVERSED**: Single Steam app with Sail + Build modes (overrides "Split project into two games") | One library entry, one Workshop, one Direct fee. Build is UI-gated, not purchase-gated. Precedents: Garry's Mod, Minecraft, Roblox. Moots bundle pricing, A-only marketing, B-demo-in-A, and which-app-hosts-Workshop questions. |
| 2026-05-27 | NPC ship budget separate from player ship budget | Player ships at Tier 2 (15 cap), NPC vessels at Tier 0/1 (~30). AI doesn't need pitch+heel fidelity; budgets don't compete. |
| 2026-05-27 | Seamless host migration on host quit | Brief-pause UX (2–5s "Host migrating…" popup). Eventually-consistent client state, snapshot broadcasts, host election, atomic authority transfer. SDR survives the migration. |
| 2026-05-27 | Steam Workshop validation: on upload AND download | Max piece count, max bounding box, max file size, structural validity (must float, must have helm + sails). Failing ships don't publish; failing downloads don't load. Specific thresholds TBD. |
| 2026-05-27 | First slice = Sea Trial Prototype (narrow Build↔Sail spine) | Build ship → save → load in Sail → float on deterministic waves → damage/overload → sink/survive. Hits risks #1 (buoyancy), #5 (ship format), and primes #2 (network determinism). See [vertical-slice.md](vertical-slice.md). |
| 2026-05-27 | Prototype engine: existing engine (lean Unreal), not engineless | Slice goal is to prove the game concept, not a custom engine. Architecture stays portable. Engineless still open for the final game — see [vertical-slice.md §3](vertical-slice.md). |
| 2026-05-27 | Web prototype landed first (single-file React + Three.js) | Faster than scaffolding an engine project. Covers slice Phases 0–6 in real 3D browser form: mesh ocean from Gerstner waves, Tier 2 sampling visualized, ship JSON, validation, cargo/damage/sinking, follow camera. Engine prototype still next. See [spikes/threejs-sea-trial/](spikes/threejs-sea-trial/). |
| 2026-05-27 | **DIRECTION RESET**: Final game is engineless native C++ — locks stack C++ / SDL3 / bgfx / Jolt / Dear ImGui / miniaudio / CMake (Steamworks later for networking) | Reverses prior "lean Unreal for prototype" path. "Engineless" = custom engine assembled from libraries, not write-every-subsystem. The Three.js browser prototype is a disposable proof spike — it proved the loop; it is not the production path. C++ chosen over Rust because Steamworks/Jolt/ecosystem are C++-native and FFI cost dominates. |
| 2026-05-28 | Multiplayer discovery: Steam Lobbies + Steamworks P2P, no custom server list for v1 | V1 uses Steam Lobbies for public/private/friends/invite session discovery and Steamworks P2P transport for small-session multiplayer. Avoids custom server-browser infrastructure and dedicated-server hosting for the first multiplayer target. Dedicated servers remain a later option if P2P proves insufficient. |
| 2026-06-05 | Dependency acquisition for Milestone 0: CMake FetchContent | Pinned deps, no manual paths, no global setup. SDL3 (native CMake), bgfx via bgfx.cmake wrapper (bundles bx+bimg; upstream bgfx is GENie/premake not CMake), Dear ImGui as a small hand-written target. Chosen over vcpkg (bgfx port freshness shaky; doesn't solve the ImGui→bgfx backend) and submodules (more ceremony; bgfx.cmake already vendors its own). Revisit vcpkg once Jolt + asset libs add many transitive deps. See [native-bootstrap-plan.md](native-bootstrap-plan.md). |
| 2026-06-05 | **SUPERSEDES the FetchContent row above** — dependency strategy: no package manager, vendor permissive no-strings libs | The FetchContent row was committed by an unexpected out-of-session commit (`186641c`, AI-authored, not issued in the working session) and was never approved. New direction: the project ships public-domain (The Unlicense — "do whatever you want"), so the repo stays self-contained with no package-manager strings. Depend only on permissive no-strings libs (all already are), vendor their source pinned, build from scratch anything with strings/cost, isolate Steamworks as optional. Exact vendoring mechanic (in-tree copy vs submodule) still open. See [native-bootstrap-plan.md](native-bootstrap-plan.md). |
| 2026-06-05 | Project license: public domain via The Unlicense | "Do whatever you want with it." Audit done first (no secrets, no PII, no vendored third-party code — spike loads CDN at runtime). Constrains deps to permissive/no-strings only (no copyleft ever) and makes Steamworks an isolated optional module so the framework stays freely usable without Steam. See [LICENSE](LICENSE) and [tech-stack.md](tech-stack.md). |
| 2026-06-05 | Native Milestone 0 complete; deps as pinned submodules; C++20 | Bootstrap done in [native/](native/): SDL3 (release-3.4.10) + bgfx (bgfx.cmake wrapper) + Dear ImGui (v1.92.8), all pinned git submodules under `extern/` — settles the vendoring mechanic (submodules, not in-tree copy or package manager). Project is C++20 (bgfx/bx headers require it). ImGui→bgfx renderer is a self-contained backend (`native/src/imgui/imgui_bgfx.*`) adapted from bgfx's own example backend + its precompiled shaders, since no official ImGui-bgfx backend exists. |
| 2026-06-05 | First 3D: ship via bgfx debugdraw, not a custom shader pipeline yet | The Test Sloop renders as oriented boxes (one per hull piece) using bgfx's debugdraw (precompiled lit shaders) — YAGNI on standing up shaderc just for boxes. debugdraw.cpp compiles directly (it only *includes* bgfx_utils.h/packrect.h, no calls). |
| 2026-06-05 | Gerstner water is CPU-displaced (wireframe grid) reusing the model's sampleWater — GPU shaders deferred | The ocean is a wireframe grid whose vertices are displaced each frame by the *same* `sea::sampleWater` the buoyancy model uses (via debugdraw drawLineList). Two wins: (1) no shaderc pipeline needed yet, and (2) the visible surface is provably identical to the buoyancy surface, so the ship will float on exactly what's drawn — no CPU/GPU divergence. GPU shader water (solid lit surface, foam) is a later fidelity/perf step, and the point where the shaderc pipeline finally gets built. |
